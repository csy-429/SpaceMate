## 3. 문제 정의

### 3.1 핵심 문제
공간 예약 플랫폼에서 (P1) 상세페이지 후기가 과도하게 많아 의사결정이 지연되고, (P2) 예약이 단발성으로 끝나 한 세션 안에서 부가가치(보완상품)가 확장되지 않는다. 결과적으로 탐색 단계 이탈이 예약 전환율(CVR)을 저해한다.

### 3.2 핵심 가치 제안
AI가 후기를 조건별로 묶어 결정을 가속하고(P1 해소), 예약 결정 직후 맥락에 맞는 보완상품을 추천해 한 흐름에서 함께 결제하게 한다(P2 해소). → 결정 가속 + 세션 가치 확장.

### 3.3 Non-Goals (이번 MVP 비목표)
- 실제 결제(PG) 연동 — 목업만
- 개인 검색이력 기반 개인화 추천 — 규제 회피 위해 페르소나 방식
- 공간→공간 추천(P2-a)
- 후기 수집 파이프라인 — 샘플 데이터로 대체
- 셀러 입점 어드민 정식화 — 시드 데이터
- 회의실·파티룸·공유주방 외 공간 유형

## 6. 유저 스토리 (User Stories)

### US-01 (후기 그룹핑)
> 공간을 고르는 사용자로서, 후기를 조건별로 묶어 보고 싶다. 빠르게 판단하기 위해서다.

- AC-1: 상세페이지에 조건 칩(예: 방음·주차·청결)이 표시된다
- AC-2: 칩을 누르면 해당 그룹의 요약이 먼저 보인다
- AC-3: 승인되지 않은 그룹은 노출되지 않는다
- AC-4: AI 미작동 시 후기 원문이 그대로 표시된다(폴백)

### US-02 (관리자 승인)
> 관리자로서, AI가 만든 후기 그룹을 검수하고 싶다. 부적절한 라벨을 거르기 위해서다.

- AC-1: 승인 대기 그룹이 큐에 표시된다
- AC-2: 승인/수정/반려가 가능하다
- AC-3: 승인된 그룹만 사용자 화면에 반영된다

### US-03 (보완상품 추천·결제)
> 파티룸을 예약하는 사용자로서, 케이크·풍선 같은 준비물을 함께 사고 싶다. 따로 사러 다니지 않기 위해서다.

- AC-1: 예약 옵션 선택 직후 보완상품이 추천된다
- AC-2: 추천 상품을 장바구니에 담을 수 있다
- AC-3: 공간+상품이 한 결제 화면에 표시된다
- AC-4: 결제는 목업으로 처리되고 실제 결제는 발생하지 않는다
- AC-5: AI 추천 실패 시 공간유형 기본 추천이 노출된다(폴백)

## 7. 비기능 요구사항 (Non-Functional Requirements)

### 7.1 성능 요구사항
- AI 후기 그룹핑 처리: 그룹 1건당 ≤10초 (비동기, 관리자 트리거)
- 보완상품 추천 응답: 사용자 대기 ≤3초, 초과 시 폴백 즉시 노출

### 7.2 보안 요구사항
- 데이터 전송 암호화: 전 구간 HTTPS
- 개인정보: 후기 내 이름·연락처는 AI 1차 필터에서 마스킹. 개인 검색이력 미수집(페르소나 방식)
- API Key 관리: OpenAI 키는 환경변수(.env)로 분리, 코드·저장소 미포함, 프론트 노출 금지(백엔드 경유 호출)

### 7.3 가용성 및 확장성 요구사항
- 장애 복구 시간: 학습용이므로 수동 재기동 허용, 목표 미설정
- LLM 장애 대응: ChatGPT 실패/타임아웃 시 하드코딩 폴백으로 추천·후기 노출 지속(서비스 무중단)
- 벡터스토어 확장: Chroma는 후기 증가 시 컬렉션 단위 확장. MVP는 단일 인스턴스

## 8. AI 특화 요구사항 (AI-Specific Requirements)

### 8.1 할루시네이션 방지 전략
- 후기 그룹핑은 입력 후기 텍스트 범위 내에서만 요약·분류하도록 프롬프트 제약(외부 사실 생성 금지)
- 관리자 승인 단계에서 부정확·부적절 라벨 최종 차단(Human-in-the-loop)
- 보완상품 추천은 시드 상품 카탈로그 후보목록을 프롬프트에 주입하고 그 범위 내에서만 추천(존재하지 않는 상품 생성 금지)

### 8.2 Fallback 전략 (서비스 장애 대응)
- 후기: AI 미작동 → 원문 표시(그룹핑 생략)
- 보완상품: AI 실패/타임아웃 → 공간유형별 하드코딩 기본 매핑 노출
- 폴백 판정은 단순 규칙(공간유형 매칭)으로 결정적 작동

### 8.3 비용·속도 예산
- 모델: GPT-4o-mini (저비용, 학습·데모용)
- 호출당 입력 ≤4K 토큰 / 출력 ≤1K 토큰, 추천은 사용자당 1회·후기 그룹핑은 관리자 배치성으로 제한
- 캐싱: 승인된 후기 그룹은 SQLite에 저장해 재호출 없이 제공(비용 절감)
- 일일 호출 상한 등은 개발단계에서 모니터링하며 조정
